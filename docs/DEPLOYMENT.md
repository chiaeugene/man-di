# Deploying Mandy — GitHub + Supabase + Render

This is a one-time setup. After it's done, every `git push` to `main` auto-deploys.

The repo is already prepared for this:
- `mandy/prisma/schema.prisma` targets Postgres (`DATABASE_URL` + `DIRECT_URL`).
- `mandy/prisma/migrations/00000000000000_init/` has the full schema as SQL, generated offline — no live database was needed to create it.
- `mandy/render.yaml` is a Render **Blueprint**: Render reads it and configures the service automatically.
- `mandy/.env.example` documents every variable needed.
- SQLite (`dev.db`) has been removed — dev and prod both run on the same kind of Postgres database now.

## 1. Push the code to GitHub

```bash
cd mandy
gh auth login              # one-time; opens a browser to authorize the CLI
gh repo create mandy --private --source=. --remote=origin --push
```

(No `gh` / prefer the website? Create an empty repo at github.com/new, then:
`git remote add origin <the URL> && git branch -M main && git push -u origin main`.)

## 2. Create the Supabase project

1. [supabase.com](https://supabase.com) → New project (pick a region close to your users, e.g. Singapore).
2. Set a strong database password — you'll need it in the connection string.
3. Once it's provisioned: **Project Settings → Database → Connection string**.
   - Copy the **Transaction** pooler URI (port `6543`) → this is `DATABASE_URL`.
   - Copy the **Direct connection** URI (port `5432`) → this is `DIRECT_URL`.
   - Both look like `postgresql://postgres.xxxx:[PASSWORD]@aws-0-region.pooler.supabase.com:PORT/postgres` — fill in the password you set.

Locally, put both into `mandy/.env` (copy from `.env.example` if you haven't), then apply the schema:

```bash
cd mandy
npx prisma migrate deploy
npm run db:seed        # optional — only for a demo tenant, skip for a clean production DB
```

Your Supabase dashboard's **Table Editor** should now show `User`, `PhotographerProfile`, `Package`, `Lead`, etc.

## 3. Deploy on Render

1. [render.com](https://render.com) → New → **Blueprint** → connect your GitHub account → select the `mandy` repo.
2. Render reads `render.yaml` and shows one service (`mandy`) ready to create.
3. Before the first deploy, fill in the secrets it asks for (these are marked `sync: false` in the blueprint, so Render prompts for them):
   - `DATABASE_URL` — the pooled Supabase URL from step 2.
   - `DIRECT_URL` — the direct Supabase URL from step 2.
   - `ANTHROPIC_API_KEY` — your Claude API key (or leave blank and switch `LLM_PROVIDER` to `openai` + set `OPENAI_API_KEY` instead).
   - `AUTH_SECRET` is auto-generated by Render (`generateValue: true`) — nothing to do.
4. Click **Apply**. Render will: `npm ci && npm run build` (which runs `prisma generate` too), then on boot run `npx prisma migrate deploy && npm run start`.
5. Once live, open the Render URL, sign up (or log in with the seeded demo account if you ran the seed step), and you're running.

### Custom domain (optional)

Render → your service → **Settings → Custom Domain** → add your domain and follow the DNS instructions (usually a `CNAME`). Auth.js will pick up the new host automatically via `AUTH_TRUST_HOST=true` — no code change needed.

## Ongoing deploys

Just `git push` to `main`. Render redeploys automatically and re-runs `prisma migrate deploy`, so any new migrations you commit get applied on the next deploy.

## Notes

- **Data persistence:** everything now lives in Supabase Postgres, which is unaffected by Render restarts, redeploys, or free-tier spin-downs — the earlier "resets on restart" risk was specifically about the old local-SQLite setup, which no longer exists.
- **Don't re-run `npm run db:seed` against a real production database** once real photographers have signed up — it's meant for demo/dev only, and while it's safe (it no-ops if `demo@mandy.my` already exists), it has no reason to run in prod.
- **WhatsApp/Instagram/Messenger (Phase 2)** aren't wired up yet — that's a separate piece of work (a webhook route + Meta app setup) layered on top of this same deployment.
